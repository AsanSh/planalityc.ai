# Архитектура: Снабжение ↔ Стройка с объектным учётом остатков

**Дата:** 2026-07-02
**Ветка:** `feat/supply-workflow-refactor` (архитектурный документ; реализация — по сабпроектам)
**Тип:** Архитектура + декомпозиция. Это **верхнеуровневый источник правды**; детальные спеки и
планы пишутся отдельно по каждой подсистеме (S0–S9).

## 1. Зачем

Цель — **точный учёт остатков материалов на объектах строительства** и сквозной процесс от
потребности прораба до списания в затраты стройки. Реальность СНГ: наперёд на центральный склад
не закупаются — покупают **по факту, по нужде, сразу на объект**. Поэтому объект (проект) —
первичное место хранения; центральный склад поддерживается, но опционален.

Сейчас фундамент уже есть в коде, но не связан в единый процесс, а логика снабжения ощущается
хаотичной (см. свёрнутую спеку рефактора).

## 2. Ключевые архитектурные решения (согласованы с заказчиком)

1. **Объект первичен, central опционален.** Склады бывают `central | project | foreman | transit`
   (уже есть). Остатки считаются и показываются по объектам; можно завести и центральный, и
   объектные склады одновременно.
2. **Единая номенклатура на всё.** `global_products` (+ категории, синонимы, прайсы поставщиков) —
   один каталог платформы, на который ссылаются и Стройка (сметы/материалы), и
   Снабжение/Склад через `global_product_id`. Нужно: (а) связать архитектурно везде, (б) наполнить
   базовым шаблоном стройноменклатуры.
3. **Двухуровневое согласование.** Заявку согласует ПТО (нужно ли и что именно), цены/оплату —
   коммерческий/финансовый директор и финчасть.
4. **Оплата через финчасть модуля Стройки.** Платёж проводится в `construction-finance` и
   привязан к проекту; снабжение не держит собственную кассу.
5. **Привязка к этапу.** Прораб в заявке указывает этап (`construction_stages`), под который берёт
   материал; списание (АВР / М-29) показывает, на какой этап и за что ушло.
6. **Приложение прораба — PWA.** Лёгкий установимый веб (manifest + service worker): создать заявку,
   принять поставку, списать, видеть остатки объекта.

## 4. Роли и что каждый видит

| Роль | Действия | Экран |
|------|----------|-------|
| **Прораб** | Создать заявку (выбрать проект+этап), принять поставку, списать, видеть остатки своего объекта | PWA прораба (S8) |
| **ПТО** | Проверить заявку, одобрить/отклонить, видеть план потребности по объектам | Веб: очередь ПТО (S9) |
| **Снабженец** | Собрать заказ (остатки объекта + изъятие с других объектов), найти поставщиков/цены, оформить договор | Веб: рабочее место снабженца (S9) |
| **Коммерч./фин. директор** | Согласовать цены/условия закупа | Веб: очередь согласования (S9) |
| **Финчасть (Стройка)** | Провести оплату по договору, привязка к проекту | Модуль Стройки → финансы (S5) |

Доступ — через существующий RBAC (`middleware/auth`, `requirePermission`, `requireRole`) +
пермишены снабжения/склада/стройки.

## 5. Сквозной воркфлоу (целевой)

Роль → действие → результат/состояние. Точные enum'ы статусов фиксируются в спеке каждой подсистемы;
единый автомат-«светофор» (`lib/supply-workflow.ts`) — источник правды по переходам.

1. **Прораб** создаёт заявку с объекта (выбирает проект, если ведёт несколько) + этап → отправляет ПТО.
2. **ПТО** проверяет и одобряет (или отклоняет). При одобрении — по матрице лимитов на сумму.
3. **Снабженец** получает заявку. Проверяет остатки **на объекте-источнике и на других объектах**;
   если нужное лежит на другом объекте — оформляет **перемещение** (изъятие). Остаток к докупке —
   собирает в заказ.
4. **Снабженец** находит поставщиков, проставляет цены; отправляет на согласование
   **коммерч./фин. директору**.
5. После согласования цен — оформляет **договор** (условия оплаты/доставки/сроки).
6. **Финчасть (Стройка)** проводит **оплату** по договору (привязка к проекту).
7. Изменённая заявка (финальный список товаров) **падает обратно прорабу**.
8. **Прораб** при поставке **подтверждает приёмку** → приход на объектный склад (партии).
9. **Списание**: материалы списываются в затраты стройки (`construction_expenses`:
   проект+этап+статья бюджета), формируются **АВР / форма М-29** с указанием этапа и назначения.
10. Остатки объекта обновляются на каждом шаге (приход/резерв/перемещение/списание).

## 6. Границы модулей (кто за что отвечает)

- **Номенклатура (platform):** канонические товары, категории, синонимы, прайсы. Не знает про объекты.
- **Склад:** физические остатки/партии/резервы/перемещения по складам (в т.ч. объектным). Ссылается
  на номенклатуру по `global_product_id`.
- **Снабжение:** заявки → согласование → сбор заказа → договор. Резервирует/списывает через Склад,
  инициирует оплату через Стройку-финансы.
- **Стройка:** проекты, этапы, бюджет, расходы, финчасть/оплата. Принимает списание материалов как
  `construction_expenses` с привязкой к этапу.

Взаимодействие — через сервисные функции, не прямыми SQL-джойнами через границу модуля.

## 7. Приложение прораба (PWA, S8)

- Отдельная лёгкая оболочка-маршрут (напр. `/foreman`) с `manifest.webmanifest` + service worker
  (кэш оболочки, устойчивость к слабой сети). Ставится на рабочий стол/телефон.
- Экраны: **Мои остатки** (по объекту), **Новая заявка** (проект+этап, позиции из номенклатуры),
  **Приёмка** (подтвердить поставку), **Списание** (в этап), **Статусы моих заявок**.
- Переиспользует существующий api-client (Orval) и компоненты; свой упрощённый навигационный shell
  по образцу `portal-shell.tsx`.

## 8. Декомпозиция на подсистемы

Каждая подсистема = отдельный цикл спека → план → реализация. Зависимости определяют порядок.

| # | Подсистема | Зависит от | Новое / достройка |
|---|-----------|-----------|-------------------|
| **S0** | Единая номенклатура: связать везде + наполнить базовым шаблоном | — | достройка + данные |
| **S1** | Объектный учёт остатков (объект первичен, central опц.) | S0 | достройка |
| **S2** | Заявка: прораб→ПТО, объект+этап, лимиты, статусы (свёрнутый рефактор) | S0,S1 | рефактор+достройка |
| **S3** | Сбор заказа: остатки объекта + изъятие с других объектов, позиции заказа | S1,S2 | достройка |
| **S4** | Согласование цен + договор снабжения | S3 | **новое** |
| **S5** | Оплата через финчасть Стройки, привязка к проекту | S4 | достройка |
| **S6** | Приёмка прорабом → приход на объект (партии) | S1,S5 | достройка |
| **S7** | Списание: `construction_expenses` + АВР/М-29 (этап, назначение) | S6 | **новое (формы)** |
| **S8** | PWA прораба (заявка/приёмка/списание/остатки) | S2,S6,S7 | **новое** |
| **S9** | Представления ролей: очереди ПТО/снабженца/согласования | S2–S5 | достройка |

## 9. Рекомендуемый порядок реализации

1. **S0 + S1 (фундамент):** единая номенклатура и объектный учёт — на них стоит всё. Без них
   остальное «висит».
2. **S2:** заявка прораб→ПТО с объектом/этапом и подключённой матрицей лимитов (вбирает свёрнутый
   рефактор `supply.ts`).
3. **S3 → S4 → S5:** сбор заказа → согласование цен/договор → оплата.
4. **S6 → S7:** приёмка → списание (АВР/М-29).
5. **S8 (PWA прораба)** и **S9 (очереди ролей)** — по мере готовности пайплайна; S8 можно начинать
   параллельно после S2/S6 в части экранов заявки/остатков.

Первый цикл брейншторм→план — **S0+S1** (наименьший, разблокирует остальное).

## 10. Влияние на уже влитый код

- `routes/supply.ts` — переструктурируется в рамках S2 (сервисный слой + тонкие роуты + единый
  автомат статусов), как описано в свёрнутой спеке.
- `supply_requests` получает обязательные `project_id` (объект) и `stage_id` (этап); статусы —
  новые значения (без обратной совместимости, согласовано).
- Warehouse-модель не ломается: добавляем удобные представления/резолверы для «остатков по объекту».
- Миграции — через self-heal список в `src/lib/migrate.ts` (последняя `0049`), только добавления и
  бэкофилл, без удаления колонок. См. `db-migration-mechanism`.
- Prod-деплой гейтится по автору коммита (`vercel-commit-author-block`).

## 11. Отложено / открытые вопросы

- **1С:** выгрузка XML списаний — ждёт образец формата конфигурации 1С заказчика (иначе структура
  угадывается). Относится к S7.
- **Точная форма М-29 / АВР:** нужен образец бланка, который использует заказчик, чтобы формы
  совпадали 1:1.
- **Офлайн PWA:** глубина офлайна (только кэш оболочки vs. офлайн-создание заявок с синхронизацией) —
  решается в спеке S8; по умолчанию начинаем с кэша оболочки.
