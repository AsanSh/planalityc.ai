# Снабжение: рефактор по сквозному воркфлоу + правки логики

**Дата:** 2026-07-02
**Ветка:** `feat/supply-workflow-refactor`
**Тип:** Рефактор структуры + точечные правки бизнес-логики (не переписыш с нуля).

## Проблема

Модуль «Снабжение / Закуп» уже влит в `main` и задеплоен (фазы 1–3 + связка каталог↔склад).
Логика рабочая, но **читается как хаос**: нет сквозного воркфлоу, статусы и правила
разбросаны, часть логики «висит в воздухе» и не подключена к процессу.

Конкретные источники хаоса, подтверждённые чтением кода:

1. **Матрица лимитов согласования не подключена.** Есть `GET /supply/approval-limits/resolve`
   (`resolveRequiredApprover(amount)`), но `POST /supply/requests/:id/approvals` его не вызывает —
   утвердить заявку может кто угодно, сумма не проверяется.
2. **Два несвязанных статус-автомата на заказе.** `supply_orders.status`
   (`draft→placed→delivered→closed`) и `supply_orders.payment_status` (7 состояний) живут
   параллельно без координации: заказ может быть `delivered` при оплате `none`, можно «закрыть»
   неоплаченный заказ.
3. **План «со склада / докупить» — тупик.** `POST /supply/requests/:id/plan` считает
   `fromStockQty/purchaseQty` и резервирует склад, но создание заказа это игнорирует. У заказа
   вообще нет позиций (`supply_order_items`), только `totalAmount`.
4. **Резерв склада течёт.** `plan` увеличивает `warehouse_stock.reserved_quantity`, но нет
   освобождения при отмене/отклонении заявки.
5. **Кредит-лимиты поставщика (`used_amount`) никто не двигает** при оплате.
6. **`POST /supply/requests` принимает любой `status`** от клиента — заявку можно создать сразу
   `approved`/`ordered` в обход процесса.
7. **`routes/supply.ts` — один файл на 698 строк**, где смешаны заявки, позиции, согласования,
   заказы, кредит-лимиты, финсогласование, оплата, матрица лимитов.

## Цель

Привести модуль к **единому читаемому сквозному воркфлоу** с единственным источником правды по
статусам, подключить висящую логику к процессу и починить перечисленные рассогласования — без
переписывания рабочих чистых либов и без раздувания объёма.

**Сквозной воркфлоу:**
`Заявка → Согласование (руководитель, по лимитам) → План (со склада / докупить) → Заказ →
Финсогласование + Оплата (финансы) → [Фаза 2: Приёмка → Склад/партии → Списание в стройку/1С]`

Согласование двухуровневое (подтверждено пользователем):
- **Заявку** согласует руководитель — «нужно ли вообще покупать», по матрице лимитов на сумму
  потребности.
- **Оплату заказа** согласуют финансы — по реальной сумме к оплате поставщику.

## Границы объёма

**В объёме этой итерации:**
- Заявка + согласование (фикс-старт, подключение матрицы лимитов, корректные переходы).
- План + заказ (связка плана с заказом, позиции заказа, заказ только на «докупить»).
- Финсогласование + оплата (координация двух автоматов заказа инвариантом, кредит-лимиты).
- Освобождение резерва склада при `rejected`/`cancelled` (течь #4).

**Вне объёма (Фаза 2, отдельный заход):**
- Глубокая связка приёмки заказа со складом/партиями (`consumeBatchesFIFO`), списание в стройку,
  выгрузка XML в 1С. Трогает `warehouse.ts` (1400 строк) — делается отдельно, чтобы не смешивать
  риски.

## Архитектура (подход A: сервисный слой + тонкие роуты + единый автомат)

```
lib/supply-workflow.ts        (+ .test.ts)   ← единственный источник правды по статусам
services/supply/
  requests.service.ts
  approvals.service.ts
  orders.service.ts
  payments.service.ts
routes/supply/
  index.ts                                   ← монтирует под-роуты
  requests.ts  approvals.ts  orders.ts  payments.ts  limits.ts
```

- **`lib/supply-workflow.ts`** — чистый модуль. Определяет разрешённые переходы заявки
  (`nextRequestStatus`) и заказа (`nextOrderStatus`), а также **инвариант** между `order.status` и
  `payment_status`. Рядом с существующей `nextPaymentStatus` (её логику переносим/оборачиваем сюда,
  чтобы был один модуль-автомат). Полностью юнит-тестируется.
- **`services/supply/*`** — оркестрация по этапам, работает в транзакциях, дёргает существующие
  чистые либы (`supply-payments`, `supply-batches`, `supply-catalog`, `warehouse-stock`). Никакого
  Express/HTTP внутри.
- **`routes/supply/*`** — тонкие HTTP-обёртки: валидация входа + вызов сервиса. Монтируются одним
  `routes/supply/index.ts`. Старый `routes/supply.ts` удаляется; точка монтирования в app
  переключается на новый индекс.
- Существующие чистые либы (`supply-payments.ts`, `supply-batches.ts`, `supply-catalog.ts`,
  `warehouse-stock.ts`) **сохраняются**; их тесты остаются зелёными.

### Изоляция и границы

Каждый сервис отвечает за один этап, общается через типизированные аргументы/результаты, тестируется
независимо. Роут не содержит бизнес-правил; автомат не содержит доступа к БД.

## Модель статусов

### Заявка (`supply_requests.status`)

```
draft → pending_approval → approved → planned → ordered → closed
                     ↘ rejected        ↘ cancelled (из любого статуса до ordered)
```

- Создание всегда в `draft` (или сразу `pending_approval` при отправке на согласование). **Клиент
  не может задать произвольный статус.**
- `approved` — только через эндпоинт согласования и только утверждающим, чья роль проходит по
  матрице лимитов для суммы заявки (`resolveRequiredApprover`).
- `planned` — после успешного разбиения «со склада / докупить».
- `ordered` — когда по части «докупить» создан заказ.
- `closed` — заявка полностью отработана (задел на Фазу 2; в этой итерации выставляется вручную/при
  закрытии заказа).

### Заказ (`supply_orders`)

Две согласованные оси со **связывающим инвариантом** в автомате:

- `status`: `draft → placed → delivered → closed`
- `payment_status`: `none → pending_finance → approved_by_finance → sent_to_payment →
  paid_partially → paid` (плюс `payment_rejected`)

**Инварианты (в `supply-workflow.ts`):**
- Нельзя `status = closed`, пока `payment_status ≠ paid`.
- Нельзя `status = delivered`, пока `payment_status` не достиг хотя бы `approved_by_finance`.
- `payment_status` двигается только событиями оплаты (`submit / approve / reject / send / pay`),
  как сейчас, но через единый модуль-автомат.

## Изменения данных (миграция 0050+, self-heal список)

Механизм миграций — через self-heal список (не `drizzle-kit generate`), см. `db-migration-mechanism`.
SQL-файл кладётся в `artifacts/api-server/drizzle/migrations/0050_supply_order_items.sql` и
регистрируется в массиве `selfHeal` в `artifacts/api-server/src/lib/migrate.ts` (последний
зарегистрированный — `0049_warehouse_item_global_link.sql`).
Только добавления и бэкофилл, **без удаления колонок** (правило безопасности БД).

1. **Новая таблица `supply_order_items`:**
   - `id`, `order_id` (FK на `supply_orders`), `request_item_id` (nullable, источник в заявке),
     `global_product_id` / `supplier_product_id` / `custom_name`, `quantity`, `unit`,
     `unit_price`, `line_total`, timestamps.
   - Заказ строится из позиций заявки с `purchase_qty > 0`. `supply_orders.total_amount` становится
     производным = Σ `line_total`.
2. **`supply_requests.status`:** новый default `draft`.
   - Бэкофилл существующих строк: `pending → pending_approval`; `approved/rejected/ordered/cancelled`
     маппятся 1:1; при необходимости `draft`/`planned` не требуют бэкофилла (новые значения).
   - Обратная совместимость строковых значений **не сохраняется** (согласовано с пользователем) —
     фронт обновляется под новые значения.

## Правки логики по этапам

| Этап | Что чиним |
|------|-----------|
| Заявка | Фиксированный старт-статус (`draft`); убрать приём `status` от клиента при создании |
| Согласование | Подключить матрицу лимитов: роль утверждающего должна быть ≥ требуемой для суммы заявки; переход писать через `nextRequestStatus` |
| План | Как сейчас (резерв склада), но **освобождать резерв** при `rejected`/`cancelled` |
| Заказ | Создание из плана: позиции = `purchase_qty`; `total_amount` = Σ позиций; запрет заказа по несогласованной заявке (уже частично есть — привести к автомату) |
| Финсогл./оплата | Координация `status`↔`payment_status` через инвариант автомата; `register-payment` двигает `used_amount` кредит-лимита поставщика |

## Обработка ошибок

- Недопустимый переход статуса → `400` с сообщением из автомата (единый формат, как сейчас в
  `applyPaymentEvent`).
- Согласование не тем, кто проходит по лимиту → `403`.
- Заказ по несогласованной заявке → `400`.
- Все мутации, затрагивающие несколько таблиц (план+резерв, заказ+позиции+статус заявки,
  оплата+кредит-лимит), — в транзакциях.
